## 五、PR 与代码管理规范

> **沿用你档案：AI 不直接 push main，主干保护 + PR Review**

### 5.1 分支策略

```
main（主干，受保护，AI 无权 push）
  ↑
  │ PR（需要王昊宇 review 通过）
  │
feature/{模块}-{功能}-{日期}
  例：feature/3-content-gen-20260708
fix/{模块}-{bug-id}
  例：fix/6-private-chat-db-read
```

### 5.2 PR 流程

```
backend-coder 完成开发
  ↓
自测通过 + 单元测试覆盖
  ↓
提交 PR（含：变更说明 / 关联 PRD / 测试截图 / 影响范围）
  ↓
Mavis 自动化检查（命名规范 / 关联文档 / 测试覆盖率）
  ↓
prd-writer review（是否符合 PRD）
  ↓
王昊宇 review（技术合理性 / 是否影响主干）
  ↓
你最终拍板（关键 PR）
  ↓
合并到 main
```

### 5.3 强制升级触发条件

- 安全漏洞修复
- 微信协议变更适配
- 平台合规要求（如 chat db 读取知情同意协议更新）
- 数据库 schema 破坏性变更

## 十、看板与产出物

### 🔴 业务链开发纪律（用户拍板，不可篡改）

> **6 模块开发必须严格按业务价值链顺序推进，不得跳跃、不得并行：**
> 
> ```
> 采撷（采集内容）→ 智造（创作内容）→ 分发（发布内容）
>                  → 获客（引流量）→ 私域（激活）→ 转化（成交）
> ```
> 
> **原因**：这是用户能感知的日耕运营闭环——
> 
> - 没有采撷就没有素材 → 智造做不出来
> - 没有智造就没有内容 → 分发没东西发
> - 没有分发就没有曝光 → 获客没流量
> - 没有获客就没有线索 → 私域没存量
> - 没有私域就没有激活 → 转化没机会
> 
> **任何人（包括 AI）不得提议调整顺序或并行多个模块开发**。如有变更需求，必须先回到你（产品决策者）拍板。
