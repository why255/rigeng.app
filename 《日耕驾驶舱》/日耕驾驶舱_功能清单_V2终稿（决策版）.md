## 十一、开发节奏（用户确认）

> **"六个模块同步开发，逐个模块开发验证进入下一个"**

这里的"同步开发"我理解为：

- **同步规划**：PRD 一次出齐 6 个模块的文档（不串行写）
- **同步架构**：系统架构、数据库设计、全局横切能力一次性设计好
- **串行实现**：一个模块做完（开发 → 自测 → 集成测试 → 用户验收）再启动下一个

### 推荐开发顺序（按**业务价值链**排，用户决策）

> **业务价值链**：**采集 → 创作 → 发布 → 获客 → 激活 → 成交**——这是用户能感知的日耕运营闭环，不得跳跃。

```
Phase 0  全局基座（架构 + 横切能力）         ←  所有模块依赖
   ↓
Phase 1  模块二 内容采撷舱                  ←  采集内容（业务链起点）
   ↓ (验证通过)
Phase 2  模块三 内容智造舱                  ←  创作内容（依赖采撷的素材）
   ↓ (验证通过)
Phase 3  模块四 全网分发舱                  ←  发布内容（依赖智造的产出）
   ↓ (验证通过)
Phase 4  模块五 精准获客舱                  ←  获客（依赖分发的曝光）
   ↓ (验证通过)
Phase 5  模块六 私域耕耘舱                  ←  激活客户（依赖获客的线索）
   ↓ (验证通过)
Phase 6  模块七 成交转化舱                  ←  成交客户（依赖私域的激活）
   ↓ (验证通过)
Phase 7  集成测试 + OTA 灰度发布
```

**顺序不可篡改的原则**：用户拍板，这个顺序就是业务价值链本身的逻辑——

- 没有采撷 → 没有弹药 → 智造做不出来
- 没有智造 → 没有内容 → 分发没东西发
- 没有分发 → 没有曝光 → 获客没流量
- 没有获客 → 没有线索 → 私域没存量
- 没有私域 → 没有激活 → 转化没机会
- 最后才能闭环到成交
