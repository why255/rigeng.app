## 一、转型决策记录

### 1.1 原 SaaS 形态的核心假设（已被推翻）

原 V7 框架隐含的部署假设：

| 维度 | 原 SaaS 假设 | 影响的功能点 |
|---|---|---|
| **部署** | 云端多租户，浏览器随时访问 | 所有模块都默认"网页访问 + 团队共享数据" |
| **账号** | 4 类角色（超管 / 运营 / 合伙人 / 推广员） | 几乎所有后台配置都依赖"管理员"视角 |
| **数据** | 云数据库统一存储 | 跨设备同步、报表、协作天然成立 |
| **算力** | 云端 GPU / 服务器 | 数字人、AI 推理跑在云端 |
| **协作** | 团队共享数据 + 实时协同 | 老师陪跑、合伙人管理、训练营都依赖"团队视图" |
| **升级** | 服务器一键升级，无感 | 不存在"用户本地版本碎片化"问题 |

### 1.2 新形态：Win exe 安装包

| 维度 | 新形态 | 关键变化 |
|---|---|---|
| **部署** | Windows 本地安装，单机运行 | 失去"浏览器随时访问"、"移动端"、"跨设备" |
| **数据** | 本地 SQLite / 本地文件 / 可选加密同步 | 隐私更好、性能更好，但失去天然协同 |
| **算力** | 本地 CPU/GPU 推理 + 云端 API 兜底 | 数字人、ASR、视频剪辑本地跑更快 |
| **微信对接** | 直接读 PC 微信本地 chat db / 协议调用 | 路径变短，但所有风险集中在客户端 |
| **平台对接** | 浏览器自动化（Playwright）做发布、评论 | 抖音/视频号/小红书封禁风险更大 |
| **升级** | 用户手动升级 / OTA 灰度 | 需要版本管理、向后兼容、降级回滚 |
| **商业模式** | 买断 / 激活码 / 增值订阅 | 与 SaaS 会员制有本质差异 |

## 二、模块增减

### 2.1 砍掉的模块

| 模块 | 理由 | 替代方案 |
|---|---|---|
| **模块一：品牌定位舱** | 由用户决定砍掉 | IP 定位环节前置：用户在首次启动 exe 时填一份"极简 IP 自填问卷"（5–8 题），作为后续内容智造舱的"风格种子"；或完全交给用户手动在系统里预设标签 |
| **模块八：数据结算舱** | 由用户决定砍掉 | 价值结算与提现逻辑由**日耕 APP 端承载**；本驾驶舱内仅做"提示入口 / 数据导出 / 跳转联动" |

### 2.2 保留的模块（6 个）

| # | 模块 | 核心价值 | Win exe 适配度 |
|---|---|---|---|
| 2 | 内容采撷舱 | 把日耕私库 + 外部爆款变成弹药库 | ⭐⭐⭐⭐ 本地爬虫 + 本地 ASR + 本地素材库 |
| 3 | 内容智造舱 | 一键把素材变精品内容 | ⭐⭐⭐⭐⭐ **最强主场**：本地数字人 + 本地剪辑 + 本地 AI |
| 4 | 全网分发舱 | 一键发到全网 | ⭐⭐ 浏览器自动化是双刃剑 |
| 5 | 精准获客舱 | 钩子 + 自动加好友 + 7 天激活 | ⭐⭐⭐ 微信协议本地可调，风控集中 |
| 6 | 私域耕耘舱 | 拟人化群发、智能互动、智能回复 | ⭐⭐⭐⭐⭐ **最强主场**：直接读微信 chat db |
| 7 | 成交转化舱 | 客户分层 + 积分进阶 + 训练营 | ⭐⭐⭐⭐ 本地跑规则引擎 + 接微信支付 |

### 2.3 被砍模块的"残留需求"处理

虽然砍了模块一/八，但有些原本属于它们的"功能点"在保留模块里仍有依赖，必须显式处理：

| 残留功能 | 原属模块 | 新归属建议 | 处理方式 |
|---|---|---|---|
| IP 人设 / 调性 / 风格 | 模块一 | 内容智造舱 | 改为"用户自填 IP 档案"轻问卷，作为风格种子 |
| 成长档案可视化（老师视角） | 模块六 + 模块八 | 个人中心 | 只展示"自己的"成长档案，不做老师陪跑 |
| 推广效果仪表盘 | 模块八 | 个人中心 | 仅保留"本人内容/获客/私域/成交"汇总，**无金额、无分成** |
| 价值计算引擎 | 模块八 | — | 彻底砍掉，结算全权交给日耕 APP |
| 提现 | 模块八 | 跳日耕 APP | 仅在驾驶舱内提供"跳转入口" |
| 活码归因追踪 | 模块八 | 全局横切 | 保留，作为线索追溯基础 |

## 三、SaaS → Win exe 转型的 7 大影响

### 3.1 架构影响

```
原 SaaS 架构                        新 Win exe 架构
─────────────                       ─────────────
[浏览器]  ──HTTPS──> [云端 Nginx]      [本地 GUI]  ──IPC──> [本地服务（Rust/Node）]
                                       │                      │
                                       │                      ├─ SQLite（业务数据）
                                       │                      ├─ 本地文件系统（素材/视频）
                                       │                      ├─ 本地向量库（Chroma / LanceDB）
                                       │                      └─ 本地模型（Ollama，可选）
                                       │                      │
                                       └─ HTTPS ──> [云端 API]
                                                  ├─ 大模型 API
                                                  ├─ 日耕 APP 积分 API
                                                  └─ 数字人/ASR 兜底（如本地 GPU 不够）
```

### 3.2 账号体系重构

- **原 SaaS**：4 类角色（超管 / 运营 / 合伙人 / 推广员），云端账号，手机号/微信登录
- **新 Win exe**：
  - 默认形态：**单机单用户**，无角色
  - 可选扩展：**主账号 + 协作子账号**（仅本机内，用系统账号登录）
  - 日耕 APP 打通：通过 API Token 绑定一次，后续静默同步积分

**→ 决策点 ①**：是否保留多角色体系？

### 3.3 跨设备 / 协同能力丢失

失去：
- 浏览器随时访问
- 手机访问
- 多设备数据同步
- 团队共享数据天然协同

获得：
- 本地性能（数字人、AI 推理、视频剪辑）
- 数据本地化（隐私好）
- 离线可用

**补偿方案**：
- 提供"加密同步盘"（用户自备 OneDrive / 坚果云）做多设备数据备份
- 团队协同改为"导出 / 导入"或"共享本地文件"

### 3.4 微信私域自动化的关键变化

| 维度 | SaaS | Win exe |
|---|---|---|
| 微信对接路径 | 云端中控 → 客户手机微信 | **本地直接读 PC 微信** |
| chat db 读取 | 受限于客户授权同步 | **直接读本地 msg.db**（高敏感） |
| 风控分散度 | 多租户分散 | **单点集中**，封号风险集中 |
| 协议稳定性 | 微信协议更新影响所有用户 | 同上，**且升级需要发版** |

**重大差异**：本地 exe 读 chat db 是 SaaS 形态做不到的"超能力"，但合规与封号风险同样集中在客户端。

### 3.5 全网分发的关键变化

- **原 SaaS**：服务端 HTTP 调用平台开放 API
- **新 Win exe**：浏览器自动化（Playwright）跑在本机
  - 风险：浏览器指纹、本机 IP、本机 Cookie 全部暴露给平台
  - 好处：可以模拟真人操作（鼠标轨迹、打字节奏）
  - 限制：视频号生态最难自动化（基本要靠半自动）

### 3.6 合规与边界（红线条目）

| 红线 | 严重度 | 处置建议 |
|---|---|---|
| 个人微信 chat db 读取 | 🔴 极高 | 必须本地化加密 + 用户强知情同意 + 文档化免责 |
| 抖音/视频号/小红书自动评论/私信 | 🔴 极高 | 建议 V1 仅"半自动 + 人工确认"，全自动作为 V2 |
| 微信群自动加好友 | 🔴 高 | 建议 V1 走"批量生成申请 + 人工点确认" |
| 多平台爬虫 | 🟡 高 | 默认只爬公开内容 + 用户授权 |
| 日耕 APP 积分打通 | 🟡 中 | 需要日耕团队确认 API 契约 |

### 3.7 商业模式重塑

| 维度 | 原 SaaS | 新 Win exe |
|---|---|---|
| 计费 | 订阅（月 / 季 / 年） | 买断 / 激活码 + 增值订阅 |
| 版本 | 无版本概念 | 标准版 / 专业版 / 旗舰版 |
| 增值 | 一价全包 | AI 算力包 / 云端 API 调用包 / 高级模板包 |
| 续费 | 自动续费 | 升级费 / 续费买断 |
| 试用 | 7 天免费试用 | 安装即可用，**部分高级功能锁定** |

## 四、优先级重排

### 4.1 原优先级（V7）

| 优先级 | 模块 |
|---|---|
| P0 | 品牌定位舱、精准获客舱、私域耕耘舱、成交转化舱 |
| P1 | 内容采撷舱、内容智造舱、全网分发舱、数据结算舱 |

### 4.2 新优先级（砍模块 + 转 Win exe 后）

| 优先级 | 模块 | 理由 |
|---|---|---|
| **P0** | 内容智造舱 | 驾驶舱的"内容大脑"；本地 exe 最强主场（数字人/剪辑/AI 推理） |
| **P0** | 私域耕耘舱 | 核心运营场景；本地 exe 可直接读微信 chat db |
| **P0** | 成交转化舱 | 变现闭环；含日耕 APP 积分打通（仍是协同关键） |
| **P1** | 内容采撷舱 | 内容弹药库；本地爬虫 + 本地 ASR |
| **P1** | 精准获客舱 | 含风控引擎；但合规风险高，部分功能需半自动 |
| **P2** | 全网分发舱 | 视频号自动化受限；优先级降到 P2，先打通公众号 + 小红书 |

**→ 决策点 ②**：全网分发舱降为 P2 是否接受？

## 五、技术选型建议

| 层 | 推荐方案 | 备选 |
|---|---|---|
| 客户端壳 | **Tauri**（Rust + WebView，体积小、性能好） | Electron（生态熟但体积大） |
| 前端框架 | Vue 3 + TypeScript + Tailwind | React |
| 后端服务（本地） | Tauri Rust 进程 + Node.js 辅助进程 | Python |
| 业务数据库 | **SQLite**（轻量、稳、文件即库） | DuckDB（分析型） |
| 向量库 | Chroma / LanceDB（本地） | Qdrant（嵌入式模式） |
| 本地大模型 | Ollama（接入 Qwen2.5 / DeepSeek） | LM Studio |
| 云端大模型 | 智谱 GLM-4 / 通义千问 / DeepSeek（兜底） | OpenAI |
| ASR | **Whisper.cpp**（本地 CPU/GPU） | 云端 ASR |
| 视频处理 | FFmpeg + Whisper + OpenCV | MoviePy |
| 数字人 | 开源 SadTalker / MuseTalk / HeyGen API | 自研 |
| 浏览器自动化 | Playwright | Selenium |
| 微信对接 | iPad/PC 协议库（合规存疑）/ 企业微信 SCRM（推荐） | — |
| 升级 | Tauri Updater + 增量更新 | 全量下载 |

**→ 决策点 ③**：技术栈是否走 Tauri？还是你已有人熟悉 Electron？

## 六、待用户拍板的 10 个关键决策

请逐项决策（✅ 同意 / ❌ 反对 / 🔧 修改建议）：

| # | 决策项 | 我的建议 | 风险 |
|---|---|---|---|
| **D1** | **数据存储**：纯本地 vs. 本地+可选云同步 | **纯本地起步**，V1.1 再加云同步 | 数据丢失风险 → 提示用户定期备份 |
| **D2** | **大模型调用**：仅云端 vs. 本地+云端混合 | **本地优先 + 云端补强**（降低成本+保护隐私） | 本地模型效果差 → 设置云端兜底开关 |
| **D3** | **角色体系**：单用户 vs. 主+子账号 | **V1 仅单用户**（砍 4 类角色） | 后续团队模式需重构 |
| **D4** | **微信私域对接**：个人微信 PC 协议 vs. 企业微信 SCRM | **个人微信 PC 协议 + SCRM 提示迁移** | 封号风险 |
| **D5** | **全网分发**：浏览器自动化 vs. 官方 API vs. 半自动+人工 | **V1 半自动 + 人工确认**（视频号最难） | 效率低 |
| **D6** | **IP 定位替代**：自填问卷 vs. 完全砍掉 | **极简自填问卷（5–8 题）** | 用户嫌烦 |
| **D7** | **价值结算**：日耕 APP 全权 vs. 驾驶舱做"提示/导出" | **日耕 APP 全权，驾驶舱仅做"跳转入口+导出 CSV"** | 体验割裂 |
| **D8** | **商业模式**：买断 vs. 订阅 vs. 双轨 | **双轨**（基础功能买断 + AI 算力订阅） | 设计复杂 |
| **D9** | **升级方式**：手动下载 vs. OTA | **OTA 自动升级 + 强制关键版本** | 老用户版本碎片 |
| **D10** | **范围裁剪**：保留核心 6 模块 vs. 大幅瘦身 | **保留 6 模块 + V1 只做每个模块的"必做项"** | 范围仍偏大 |

## 七、范围与节奏建议

### 7.1 V1（建议 8–10 周）只做"必做项"

> 每一项的"必做/可做/不做"边界，会在《功能清单 V2》里逐项标 ✅/⚠️/❌。

**必做（V1 上线即可用）**：
- 内容智造舱：标题 / 图文 / 长文 / 短视频脚本 / 朋友圈文案 + 品牌语校验 + 铁律
- 私域耕耘舱：标签管理 + 拟人化群发 + 关键词触发 + 智能回复 + 人工介入提醒
- 成交转化舱：客户分层 + 产品库 + 自动推送 + 微信支付对接 + 训练营基础版
- 内容采撷舱：日耕私库 API + HR 模块分类 + 视频 ASR 转写 + 素材库
- 精准获客舱：钩子模板 + 活码归因 + 加好友规则 + 风控引擎 + 自动发福利 + 自动打标签

**可做（V1.1，2–3 周补完）**：
- GEO 优化 / 智能数字人 / 智能配图 / 视频自动剪辑 / 数据报表

**不做（V2+ 评估）**：
- 自动评论 / 自动私信 / 微信群导出 / 手机号自动加好友（全自动）
- 老师协同工作台（已隐含砍掉）
- 团队业绩排行（已隐含砍掉）

### 7.2 一句话总结

> **"砍两头、保中间、聚焦本地能力"** ——砍掉 IP 定位和数据结算，保住内容智造 / 私域耕耘 / 成交转化三大核心主场，用本地 exe 的能力优势对冲 SaaS 形态的天然协同优势。
