## 🔴 三条铁律（不可篡改）

### 1. 业务链开发纪律（用户拍板）

> **6 模块开发必须严格按业务价值链顺序推进，不得跳跃、不得并行开发：**
> ```
> 采撷（采集内容）→ 智造（创作内容）→ 分发（发布内容）

> ```

## 四、Phase 2–7 · 模块串行开发（每模块 4–12 工作日）

> **关键约束**：一个模块做完（开发 → 自测 → 集成测试 → 用户验收）再启动下一个。

### 4.1 推荐开发顺序（按"价值密度 + 依赖"）

| 顺序          | 模块        | 理由                                            | 估算        |
| ----------- | --------- | --------------------------------------------- | --------- |
| **Phase 2** | 模块三 内容智造舱 | 最强主场、最有价值、最早出可见 demo 给用户                      | 8–10 工作日  |
| **Phase 3** | 模块六 私域耕耘舱 | 最强主场、内容消耗、依赖模块三的内容输出                          | 10–12 工作日 |
| **Phase 4** | 模块五 精准获客舱 | 流量入口、独立性较强                                    | 8–10 工作日  |
| **Phase 5** | 模块七 成交转化舱 | 变现闭环、依赖前面模块的客户/标签 | 6–8 工作日   |
| **Phase 6** | 模块二 内容采撷舱 | 内容弹药库、独立性较强                                   | 4–6 工作日   |
| **Phase 7** | 模块四 全网分发舱 | 内容放大、依赖所有内容产出                                 | 4–6 工作日   |

### 4.2 每个 Phase 的标准动作

```
开发启动会议（0.5 天）
  ├── 重读 PRD + 原型
  ├── backend-coder 拆分任务 / 排期
  └── 风险点列举

开发实施（N 天）
  ├── 编码（backend-coder 主，prd-writer 答疑）
  ├── 单元测试（覆盖核心路径）
  └── 内部 Code Review

集成测试（0.5–1 天）
  ├── 与全局横切能力集成
  ├── 跨模块联调
  └── 性能压测（数字人/ASR/视频处理）

用户验收（0.5–1 天）
  ├── 部署到测试机
  ├── 用户现场操作核心场景
  ├── 记录反馈 / bug / 改进项
  └── 拍板通过 or 回炉

下一模块启动
```

### 4.3 每模块交付物

- 源代码（含 git tag）
- 单元测试报告
- 集成测试报告
- 用户验收纪要
- 该模块的接口契约（与全局 / 跨模块）
- 更新后的《功能清单 V2 终稿》（如有调整）

## 五、Phase 8 · 集成测试 + OTA 灰度（3–5 工作日）

### 5.1 集成测试

- 6 模块端到端流程跑通
- 真实账号 / 真实微信 / 真实平台账号的完整链路测试
- 数据一致性、并发安全、本地数据库事务完整性

### 5.2 OTA 灰度方案

- 灰度策略：小范围 → 50% → 100%
- 强制升级触发：涉及合规 / 安全的版本强制升级
- 回滚方案：上一稳定版本保留 30 天可降级
- 升级日志：本地日志 + 匿名升级统计回传（可选）

### 5.3 上线检查清单

- [ ] 数字人质量达标（2K 分辨率 / 口型自然 / 表情自然）
- [ ] 风控引擎稳定运行 ≥ 7 天
- [ ] chat db 知情同意协议落实
- [ ] 日耕 APP 积分打通联调通过
- [ ] 商业模式（买断 + AI 算力订阅）计费链路打通
- [ ] OTA 升级方案演练通过
- [ ] 隐私政策 / 用户协议法务 review
- [ ] 备份恢复演练通过
