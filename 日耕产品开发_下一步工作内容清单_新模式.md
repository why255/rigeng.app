# 第一阶段：全局系统性规划

**阶段目标：** 一次性覆盖13个子模块的全局技术架构、统一数据模型、标准接口规范和前端组件库规划。  
**核心理念：** 先画好完整的蓝图，再动手盖楼，避免盖到三楼发现地基要返工。

## 步骤 1：全局技术架构设计

### 1.1 做什么
由WorkBuddy + Claude Code协同，基于四板块需求洞察文档，设计覆盖全部13个子模块的技术架构。

### 1.2 怎么做

| 任务 | 执行者 | 输入 |
|------|--------|------|
| 读取13个子模块需求 | WorkBuddy + Claude Code | 四板块需求洞察文档 |
| 提取每个模块的技术需求 | Claude Code | 上述文档 |
| 设计系统分层架构 | Claude Code | 技术需求汇总 |
| 确定技术栈 | Claude Code + 昊宇 | 技术选型评估 |
| 输出架构文档 | WorkBuddy | Claude Code的架构设计 |

**AI执行方式：**
- Claude Code读取全部四份需求洞察文档，AI自动提取每个子模块的技术关键点（语音、存储、检索、AI推理、实时协作、甘特图等）
- Claude Code设计分层架构：接入层→业务服务层→数据中台→基础设施层
- WorkBuddy将架构设计整理为标准文档格式

### 1.3 注意事项
- **必须逐模块精读需求洞察**，不能只看摘要。每个模块的特殊需求要单独标注：
  - 朝有规划：离线语音、方言支持、30秒静默守护
  - 情绪树洞：本地加密存储、暗色模式、危机干预≤3秒响应
  - 智能记录：1小时录音≤1GB、声纹识别（二期）
  - 智能办公：多人实时协作编辑、模块间数据自动打通
  - 高维求职：企业情报采集（老师后台独立入口）
- **采用"80%通用+20%预留"原则**：通用能力（用户管理/知识库/语音/搜索）一次性设计到位，特殊能力（情绪树洞本地加密/企业情报采集流程）预留扩展接口
- **技术栈需与昊宇确认**：确认服务器环境支持选定的数据库、缓存、消息队列等中间件
- 架构设计要支持**模块化微服务**：每个基础服务独立部署，单个服务变更不影响全局

### 1.4 输出成果
- 《全局技术架构文档》：系统分层图、技术栈选型、模块划分、部署拓扑
- 《技术风险清单》：高风险项+备选方案

## 步骤 2：统一数据模型设计

### 2.1 做什么
基于13个子模块的数据流关系，设计统一的数据模型，确保跨模块数据天然畅通。

### 2.2 怎么做

| 数据域 | 覆盖模块 | 由谁设计 |
|--------|---------|---------|
| 用户域 | 全部13模块 | Claude Code |
| 知识库域 | 全部13模块（公私智库为核心） | Claude Code |
| 对话域 | 朝有规划/暮有复盘/情绪树洞/智能问答 | Claude Code |
| 记录域 | 智能记录 | Claude Code |
| 文档域 | 智能办公/高维求职/品牌打造 | Claude Code |
| 项目域 | 交付一笔订单 | Claude Code |
| 分析域 | 数据分析 | Claude Code |
| 权限域 | 全部13模块 | Claude Code |

**AI执行方式：**
- Claude Code基于需求洞察中的"输出与数据流"章节，提取每个模块的产出物格式
- Claude Code设计统一文档模型：一份SOP在朝有规划是"复盘成果"，在公私智库是"归档文档"，在智能问答是"引用源"——底层是同一份数据，只是展示层不同
- Claude Code设计数据版本化机制，支持字段增删不影响存量数据

### 2.3 注意事项
- **文档统一模型是核心设计**：智能记录的萃取报告、暮有复盘的SOP、智能问答的沉淀手册、高维求职的技能晶体——本质上都是"结构化文档"，底层用同一套模型，通过类型标签和模块归属区分
- **音频/文件存储独立设计**：录音、PPT、PDF等大文件走对象存储，元数据走关系数据库
- **情绪树洞隐私数据必须分层**：本地加密缓存（不上传）+ 云端脱敏存储（用于分析），这个分层在数据模型层面就要定义清楚
- **预留企业情报的采集数据模型**：虽然高维求职在第四波才开发，但情报采集的数据结构（来源/采集时间/审核状态/交付状态）要在统一模型中预留

### 2.4 输出成果
- 《统一数据模型文档》：ER图、实体定义、字段说明、关联关系、索引设计
- 《数据流转图》：13模块间的数据流向、归档路径、调用关系
- 《数据隐私分级表》：哪些数据本地存储、哪些加密上传、哪些脱敏处理

## 步骤 3：标准接口规范设计

### 3.1 做什么
定义模块间通信的标准接口协议，确保13个模块可以无缝对接。

### 3.2 怎么做

**核心接口清单（AI自动生成）：**

| 接口类别 | 接口示例 | 调用方→提供方 |
|---------|---------|-------------|
| 归档接口 | saveToKnowledgeBase(doc) | 朝有规划/暮有复盘/智能记录→公私智库 |
| 检索接口 | searchKnowledgeBase(query, filters) | 智能问答/智能办公→公私智库 |
| 行动项同步 | syncActionItems(items, targetModule) | 智能记录→朝有规划 |
| 语音处理 | processVoice(audio, options) | 朝有规划/暮有复盘/情绪树洞/智能记录→语音引擎 |
| AI推理 | generateResponse(context, model) | 全部对话模块→AI引擎 |
| 数据分析 | getAnalytics(userId, metrics, period) | 数据分析→全部模块 |
| 消息推送 | sendNotification(userId, message, channel) | 全部模块→消息服务 |
| 文件存储 | uploadFile/ downloadFile | 智能记录/智能办公→存储服务 |

**AI执行方式：** Claude Code基于数据模型设计，自动生成RESTful API规范（路径/方法/参数/返回值/错误码）。

### 1.3 注意事项
- **接口设计遵循"一次定义，全局复用"**：归档接口在朝有规划调用是"工作计划归档"，在暮有复盘调用是"SOP归档"，在智能记录调用是"萃取报告归档"——同一个接口，不同参数
- **错误码全局统一**：不要每个模块自己定义错误码，全局一套错误码体系
- **接口版本化**：预留v1/v2版本号，后续升级不影响存量调用
- **每个接口配套自动生成测试用例模板**：后续模块开发时直接复用

### 3.4 输出成果
- 《标准接口规范文档》：全部接口的定义（路径/方法/参数/返回值/错误码）
- 《接口测试用例模板》：每个接口的测试用例标准格式

# 第二阶段：统一后台框架搭建

## 步骤 7：项目骨架搭建与开发环境配置

### 7.1 做什么
Claude Code搭建项目骨架，昊宇配置开发环境。

### 7.2 怎么做

| 任务 | 执行者 | 内容 |
|------|--------|------|
| 项目初始化 | Claude Code | 按模块化微服务架构创建目录结构 |
| 数据库初始化 | Claude Code | 按统一数据模型创建全部表结构 |
| 基础框架搭建 | Claude Code | API框架、ORM、缓存、消息队列接入 |
| 开发环境配置 | 昊宇 | 本地开发环境、CI/CD流水线、代码仓库分支策略 |
| API网关配置 | Claude Code | 统一入口、路由、鉴中间件 |

**AI执行方式：** Claude Code基于《全局技术架构文档》和《统一数据模型》，AI自动生成项目骨架、数据库迁移脚本、基础配置代码。

### 7.3 注意事项
- **数据库表结构一次性创建全部**：13个模块的表一次性定义，后续只增数据不调整结构
- **代码模块化**：每个基础服务独立目录，后续模块的业务代码也独立目录，接口清晰
- **README必须完整**：项目结构说明、启动命令、环境变量、数据库连接方式——昊宇能不看代码就理解项目
- **Git分支策略确定**：main（生产）、develop（集成）、feature/模块名（开发）
- **环境分离**：开发环境/测试环境/生产环境完全分离，配置不硬编码

### 7.4 输出成果
- 可运行的项目骨架（git仓库可克隆、可启动）
- 数据库迁移脚本（可执行、可回滚）
- 项目README文档
- 开发环境就绪确认

## 步骤 8：八大基础服务开发

### 8.1 做什么
Claude Code开发8大基础服务，覆盖13个模块的全部基础能力需求。

### 8.2 怎么做

| 服务 | 功能范围 | 覆盖模块 |
|------|---------|---------|
| **① 用户/权限服务** | 用户注册/登录、VIP等级、权限控制（老师/学员/运营） | 全部13模块 |
| **② 公私知识库服务** | 文档归档/分类/检索、RAG语义搜索、私有库+携君库+互联网三源 | 全部13模块 |
| **③ 语音/AI引擎服务** | 语音识别（在线+离线）、AI推理调用、多轮对话管理 | 朝有规划/暮有复盘/情绪树洞/智能记录/智能问答 |
| **④ 文件存储服务** | 音频上传/压缩（1h≤1GB）、文档存储、大文件管理 | 智能记录/智能办公/高维求职 |
| **⑤ 搜索/RAG服务** | 语义检索、TopN召回、关键词补充搜索、命中率监控 | 全部13模块 |
| **⑥ 消息/推送服务** | App推送、短信、连续未使用提醒、危机干预通知 | 朝有规划/暮有复盘/情绪树洞 |
| **⑦ 数据仪表盘服务** | 埋点采集、指标计算、可视化数据接口 | 数据分析 |
| **⑧ 安全/加密服务** | 数据加密、本地存储加密、脱敏处理、隐私合规 | 情绪树洞/全部涉及敏感数据的模块 |

**AI执行方式：** Claude Code基于《标准接口规范》，逐个服务开发。每个服务包含：API接口实现、数据库操作、单元测试、API文档。AI按优先级顺序开发，先开发被最多模块依赖的服务（用户/权限→知识库→语音/AI→其他）。

### 8.3 注意事项
- **服务间通过标准API通信**，不直接操作对方数据库
- **每个服务独立单元测试**，Claude Code自测通过
- **知识库服务是核心中的核心**：必须同时支持结构化文档存储和向量语义检索，为13个模块提供统一的归档和查询能力
- **安全服务必须覆盖情绪树洞的本地加密需求**：即使情绪树洞在第三波才开发，加密/脱敏的基础设施要在第二波就可用
- **语音引擎要预留方言包扩展接口**：第一期只支持普通话，但架构要支持后续加载四川话/广东话方言包
- **每个服务开发完成后，Claude Code自测 → 昊宇代码审查 → 通过后才开发下一个服务**

### 8.4 输出成果
- 8大基础服务全部可运行
- 每个服务的API文档
- 每个服务的单元测试（全部通过）
- 服务间联调测试通过

- **方案版本管理铁律**：输出A/B两版（差异为"交付方式略有不同"），**绝对禁止**命名中出现"基础版/标准版/高级版"。由老师和用户共同确定最终交付版本
- **三源智能调用**：携君知识库（方法论/模板/案例）+ 私有知识库（用户已有经验/SOP）+ 智能办公各模块。三源调用可用率100%
- **方案跨客户复用**：A客户方案的框架和诊断方法论可快速改编给B客户，越做越轻松。方案复用引擎（V2.2新增）
- **老师带着做不是替做**：2-5次内置视频辅导，老师传授方法论+逐模块打磨（确认逻辑/优化表达/补充遗漏），确保用户自己掌握方法论可复用至下一个客户
- **客户文档隐私铁律**：用户上传的客户企业文档（组织架构/薪酬数据等）仅存储于私有知识库，不用于模型训练，不上传公有云
- **制度升级可追溯**：新方案末尾自动标注"本方案参照了原XX制度第X条第X款，在此基础上做了以下升级：1.……2.……"
- **方案与初始报价基本一致**：报价可微调，最终由老师和用户共同确定
- **方案定稿自动归档私有库→手动导入交付订单**：方案中的交付节点/里程碑自动拆解为甘特图任务
